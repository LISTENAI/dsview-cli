# Milestone v1.2: DSView protocol decode CLI foundation

**Status:** ✅ SHIPPED 2026-04-22
**Phases:** 14-17
**Total Plans:** 12

## Overview

This milestone turned protocol decode from a research direction into a shipped headless CLI workflow. It added a separate decode runtime around `libsigrokdecode4DSL`, a JSON config and validation model, offline decode execution over raw logic artifacts, and a final machine-readable reporting contract that stays separate from `capture`.

## Phases

### Phase 14: Decode Runtime Boundary and Decoder Registry

**Goal**: Expose `libsigrokdecode4DSL` through the Rust/native boundary and make decoder metadata inspectable from the CLI.
**Depends on**: Phase 13 from milestone `v1.1`
**Plans**: 3 plans

Plans:
- [x] 14-01: Map the `libsigrokdecode4DSL` API and packaging requirements into a minimal native decode runtime boundary.
- [x] 14-02: Implement Rust-owned decoder registry and inspect structures over the new runtime seam.
- [x] 14-03: Add `decode list` / `decode inspect` CLI commands with automated contract coverage.

**Details:**
- Added a sibling decode runtime (`dsview_decode_runtime`) so decoder discovery stays isolated from the capture runtime.
- Exposed canonical upstream decoder metadata to Rust and the CLI without introducing a second decoder-id namespace.
- Shipped `decode list` and `decode inspect <decoder-id>` with JSON-first contracts and readable text renderers.

### Phase 15: Decode Config Model and Validation

**Goal**: Create a config-driven decoder stack model that stays aligned with DSView concepts and validates before runtime execution.
**Depends on**: Phase 14
**Plans**: 3 plans

Plans:
- [x] 15-01: Define the decode config schema and DSView-compatible decoder-stack model in Rust.
- [x] 15-02: Implement metadata-driven config validation and stable validation errors.
- [x] 15-03: Add CLI config loading/diagnostics coverage for valid and invalid decode configs.

**Details:**
- Added a JSON-first decode config contract with canonical upstream ids, numeric channel bindings, typed option values, and a linear stack model.
- Implemented strict pre-execution validation for schema, metadata, option typing, and stack compatibility.
- Shipped `decode validate --config <PATH>` as a validation-only CLI surface.

### Phase 16: Offline Decode Execution

**Goal**: Run DSView protocol decoders against saved logic artifacts from the CLI, including stacked decoder flows.
**Depends on**: Phase 15
**Plans**: 3 plans

Plans:
- [x] 16-01: Define the saved-artifact input contract and native sample-feeding path for offline decode.
- [x] 16-02: Implement decode-session execution and stacked decoder orchestration through Rust/core/native layers.
- [x] 16-03: Add regression coverage for offline decode success and representative execution failures.

**Details:**
- Defined raw logic artifacts as the canonical decode input rather than VCD text.
- Implemented chunked session feeding with absolute sample numbering and strict linear stack execution.
- Shipped `decode run` for validated configs and raw logic input artifacts.

### Phase 17: Decode Output and Workflow Reporting

**Goal**: Finalize decode output contracts, artifact reporting, and stable failure taxonomy for a separate decode workflow.
**Depends on**: Phase 16
**Plans**: 3 plans

Plans:
- [x] 17-01: Define the stable decode output schema and annotation event model.
- [x] 17-02: Implement output/artifact writing plus stable workflow/reporting errors.
- [x] 17-03: Lock CLI output contracts and end-to-end decode reporting coverage.

**Details:**
- Finalized a canonical `run + flat events` machine-readable output schema.
- Added stable, fine-grained failure reporting with optional partial diagnostics on failures while keeping top-level status binary.
- Added optional `--output` artifact persistence that writes the same canonical JSON document used by stdout.

## Milestone Summary

**Key Decisions:**
- Keep decode as a separate workflow from `capture` and preserve a config-driven interface.
- Treat canonical upstream decoder/channel/option ids as the source of truth end to end.
- Use raw logic artifacts, not VCD text, as the canonical offline execution input.
- Keep user-facing execution status binary even when partial diagnostics are available.

**Issues Resolved:**
- Closed the missing decode runtime symbol-visibility bug by loading the decode runtime with globally visible Python symbols.
- Reconciled the Phase 17 verifier gap by restoring the plan-declared partial-diagnostics regression test name.

**Issues Deferred:**
- Capture+decode orchestration remains deferred.
- Public `partial_success` semantics remain deferred.
- Broader hardware support and GUI parity remain out of scope.

**Technical Debt Incurred:**
- `DSView` submodule local fixes are still managed as local patches rather than upstreamed changes.
- Some upstream decoder warnings from vendored scripts remain acceptable-but-noisy rather than fully cleaned up in this milestone.
