<objective>
Define and generate the JSON metadata sidecar for successful exports so downstream automation can trust both what was captured and what artifacts were produced.
</objective>

<scope>
This plan covers metadata schema definition, actual-versus-requested field handling, path derivation, file-write ordering, and CLI-visible reporting for complete artifact success. It does not add the final golden-file corpus or complete manual sign-off matrix from the next plan.
</scope>

  <name>Lock the v1 metadata schema and field semantics</name>
  <read_first>
    - .planning/phases/05-export-artifacts/05-RESEARCH.md
    - .planning/REQUIREMENTS.md
    - crates/dsview-core/src/lib.rs
  </read_first>
  <files>.planning/phases/05-export-artifacts/05-RESEARCH.md, crates/dsview-core/src/lib.rs</files>
  <action>Translate the Phase 5 research recommendations into a concrete versioned metadata schema and document how each field is sourced. Make the distinction between requested configuration, observed acquisition facts, and final artifact facts explicit so later automation can trust the sidecar contract.</action>
  <acceptance_criteria>
    - the schema includes `schema_version`, tool identity, capture timestamp, device model, enabled channels, sample rate, requested sample limit, actual sample count, and artifact paths
    - `EXP-04` fields are mapped to a concrete source of truth in code rather than implied from request objects alone
    - timestamps are defined in UTC and serialized in a stable RFC 3339-compatible form
    - numeric facts such as sample rate and sample counts are represented as numeric JSON values rather than strings
    - the schema leaves room for later extension without breaking v1 consumers
  </acceptance_criteria>
</task>

<task type="auto">
  <name>Enforce artifact path derivation and write ordering for complete v1 success</name>
  <read_first>
    - crates/dsview-core/src/lib.rs
    - crates/dsview-cli/src/main.rs
    - .planning/phases/05-export-artifacts/05-RESEARCH.md
  </read_first>
  <files>crates/dsview-core/src/lib.rs, crates/dsview-cli/src/main.rs</files>
  <action>Define the Phase 5 output-path contract now so Phase 6 can polish it rather than redesign it. Derive the metadata path deterministically from the VCD output path, write metadata only after VCD export succeeds, and require temp-file-plus-promotion or equivalent cleanup-safe behavior so incomplete artifacts are not mistaken for final success.</action>
  <acceptance_criteria>
    - metadata path derivation from the chosen VCD path is deterministic and documented in code-facing tests
    - metadata is written strictly after successful VCD export
    - a metadata write failure causes the overall command to fail because the promised v1 artifact set is incomplete
    - metadata uses an atomic or cleanup-safe finalization path so failed writes do not leave a misleading final-path sidecar
    - VCD success assumptions rely on the temp-file/finalization contract established in `05-01`, not on best-effort direct final-path writes
  </acceptance_criteria>
</task>

  </acceptance_criteria>
