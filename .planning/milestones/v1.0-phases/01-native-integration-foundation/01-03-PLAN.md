<objective>
Add the smallest meaningful automated smoke coverage for the selected native integration path so the project can detect broken headers, linkage, or initialization before Phase 2 starts building device workflows.
</objective>

<scope>
This plan adds smoke validation only. It should not require real device capture or protocol decode.
</scope>

<task type="auto">
  <name>Create a native smoke test target</name>
  <read_first>
    - crates/dsview-sys/src/lib.rs
    - crates/dsview-sys/Cargo.toml
    - .planning/phases/01-native-integration-foundation/01-RESEARCH.md
    - DSView/libsigrok4DSL/tests/check_core.c
  </read_first>
  <files>crates/dsview-sys/tests or crates/dsview-sys/src/lib.rs</files>
  <action>Create an automated smoke test or minimal test binary that exercises the narrowest proven native path, preferably initialization and teardown such as `sr_init` and `sr_exit`, or the equivalent smallest stable API calls exposed by `dsview-sys`. Keep the test free of hardware requirements and GUI launch assumptions.</action>
  <acceptance_criteria>
    - a test file exists under `crates/dsview-sys/tests` or equivalent test module exists in `crates/dsview-sys/src/lib.rs`
    - the smoke target invokes at least one real native entry point from the chosen boundary
    - the smoke target does not require a connected device to pass
  </acceptance_criteria>
</task>

<task type="auto">
  <name>Wire smoke coverage into the Rust workflow</name>
  <read_first>
    - Cargo.toml
    - crates/dsview-sys/Cargo.toml
    - smoke test source created in the previous task
  </read_first>
  <files>Cargo.toml, crates/dsview-sys/Cargo.toml</files>
  <action>Ensure the smoke test runs through the normal Rust workflow, using `cargo test -p dsview-sys` or a similarly narrow command. Keep the command scoped so failures identify the native boundary directly rather than the entire future application surface.</action>
  <acceptance_criteria>
    - `cargo test -p dsview-sys` is the documented smoke command or an equivalent narrow command is clearly encoded
    - test configuration limits scope to the sys/native boundary
  </acceptance_criteria>
</task>

  <acceptance_criteria>

    - documentation lists at least two revalidation triggers tied to submodule or build-boundary changes
    - documentation states that device discovery and capture behavior remain out of scope for Phase 1 smoke coverage
  </acceptance_criteria>
