<objective>
Extend the native integration boundary so Rust can initialize the DSView-backed runtime, inspect discovered devices, and tear down cleanly while still keeping unsafe details isolated in `dsview-sys`.
</objective>

<scope>
This plan is limited to the native/sys boundary. It does not add safe Rust orchestration in `dsview-core`, user-facing CLI commands, capture execution, or export.
</scope>

  <files>crates/dsview-sys/wrapper.h, crates/dsview-sys/src/lib.rs, repo-owned shim source</files>
  <action>Choose the narrowest workable Rust-facing boundary for Phase 2 around the DSView `ds_*` facade. Prefer a tiny repo-owned shim that wraps only the operations needed for init, resource-path configuration, device list retrieval, active-device open, active-device release, and last-error lookup. Keep raw types and unsafe declarations confined to `dsview-sys`.</action>
  <acceptance_criteria>
    - the chosen boundary uses DSView frontend/facade entry points rather than DSView GUI classes
    - `crates/dsview-sys/src/lib.rs` exposes only raw boundary items and no safe orchestration logic
    - the plan documents how `DSLogic Plus` support is identified from upstream profile evidence
  </acceptance_criteria>
</task>
