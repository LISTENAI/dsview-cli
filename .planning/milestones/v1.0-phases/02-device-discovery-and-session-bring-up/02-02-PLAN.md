<objective>
Build the safe Rust-side device and session bring-up layer so later CLI work can discover `DSLogic Plus`, open a session, and release it cleanly without handling raw native state directly.
</objective>

<scope>
This plan adds Rust orchestration only. It does not expose CLI commands, start captures, configure acquisition parameters, or export artifacts.
</scope>

<task type="auto">
  <name>Implement safe lifecycle orchestration</name>
  <read_first>
    - crates/dsview-sys/src/lib.rs
    - crates/dsview-core/src/lib.rs
    - Phase 2 research notes
  </read_first>
  <files>crates/dsview-core/src/session.rs, crates/dsview-core/src/error.rs, crates/dsview-core/src/lib.rs</files>
  <action>Wrap the sys boundary in a safe API that initializes the library once, lists devices, opens a selected supported device, and guarantees release/teardown on success, failure, and drop. Keep single-owner behavior explicit if the underlying DSView runtime uses process-global state. Make resource-directory resolution and validation an explicit part of the bring-up contract before operations that may require firmware assets.</action>
  <acceptance_criteria>
    - open/close flow is safe to call from Rust without leaking resources
    - the API shape makes it hard to forget teardown
    - the core layer owns validation and propagation of the firmware/resource directory before bring-up
    - global runtime assumptions are documented in code comments or module docs where needed
  </acceptance_criteria>
</task>
