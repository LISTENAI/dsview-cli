<objective>
Add the validation coverage and manual UAT needed to trust Phase 4 acquisition execution on `DSLogic Plus`, with special attention to the explicit clean-capture success rule, local preflight readiness, rerunability, incomplete-run detection, and operator-facing diagnostics.
</objective>

<scope>
This plan covers automated tests and manual hardware validation for acquisition lifecycle behavior only. It does not validate VCD output, metadata artifacts, or end-to-end productization work from later phases.
</scope>

<task type="auto">
  <name>Add core orchestration tests for clean, failed, and incomplete runs</name>
  <read_first>
    - crates/dsview-core/src/lib.rs
    - .planning/phases/04-acquisition-execution/04-RESEARCH.md
    - .planning/REQUIREMENTS.md
  </read_first>
  <files>crates/dsview-core/tests/acquisition.rs</files>
  <action>Create focused tests around synthetic acquisition summaries and mocked session outcomes. Cover the codified clean finite-capture rule, start failure, terminal event error, detach, incomplete packet or end-marker state, timeout, cleanup failure, and preflight-blocked execution so the core layer's success and failure contract is proven without real hardware.</action>
  <acceptance_criteria>
    - tests assert that success requires the exact terminal-event and packet-completeness combination codified in Phase 4 research
    - tests assert that missing logic packets or missing end-of-stream markers map to failure when the success rule requires them
    - tests assert that cleanup failure overrides an otherwise clean run into a command failure
    - tests assert distinct mapping for preflight-blocked, start failure, run failure, detach, incomplete, timeout, and cleanup-failure categories
    - no default core test requires a connected logic analyzer
  </acceptance_criteria>
</task>

  <acceptance_criteria>
    - manual steps call out the current `LIBUSB_ERROR_ACCESS` risk and permission prerequisite clearly
    - manual UAT begins with preflight checks for USB permissions, runtime resource paths, and open/config-apply readiness on this machine
    - manual validation covers one clean finite capture and explicitly checks it against the codified success rule rather than only checking exit code `0`
    - manual validation covers both success and failure recovery, not just initial capture start
    - the plan explicitly verifies that the device is reusable after both successful and failed runs
    - the validation notes state which failure classes are manual-only, synthetic-only, or environment-gated so no promised category is left unverified
    - validation notes do not assume standalone capability or config inspection commands exist outside the run flow
  </acceptance_criteria>
</task>
