<objective>
Strengthen the acquisition path so failed, incomplete, timed-out, or locally blocked runs return clear non-zero results and still leave the DSView runtime, active session, and `DSLogic Plus` device in a reusable state.
</objective>

<scope>
This plan covers cleanup ordering, failure classification, bounded-wait policy, and CLI-visible diagnostics for acquisition execution. It does not add export artifacts or broader CLI workflow polish from later phases.
</scope>

<task type="auto">
  <name>Define terminal acquisition outcome categories and diagnostic mapping</name>
  <read_first>
    - .planning/phases/04-acquisition-execution/04-RESEARCH.md
    - crates/dsview-core/src/lib.rs
    - crates/dsview-cli/src/main.rs
    - Phase 2 error normalization patterns
  </read_first>
  <files>crates/dsview-core/src/lib.rs, crates/dsview-cli/src/main.rs</files>
  <action>Map acquisition results into stable domain categories that distinguish immediate start failure, terminal event error, device detach, incomplete packet or end state, timeout or bounded-wait expiry, cleanup failure, and preflight-blocked execution. Preserve native `last_error` context and event reason where available, but keep the CLI contract centered on actionable stable error kinds and non-zero exit behavior.</action>
  <acceptance_criteria>
    - failures that need different operator action are surfaced as distinct categories
    - incomplete packet or end-marker state is treated as failure rather than silent success
    - cleanup failure after an otherwise clean capture still causes the command to fail because `RUN-02` requires device reusability
    - preflight-blocked execution is distinguishable from a true acquisition-start or run failure
    - CLI diagnostics can render both human-readable guidance and machine-oriented structured fields without inventing Phase 6 UX
  </acceptance_criteria>
</task>

<task type="auto">
  <name>Implement deterministic stop and release ordering for all acquisition paths</name>
  <read_first>
    - crates/dsview-core/src/lib.rs
    - crates/dsview-sys/src/lib.rs
    - crates/dsview-sys/bridge_runtime.c
    - Phase 4 research notes on `ds_stop_collect()` and `ds_release_actived_device()` behavior
  </read_first>
  <files>crates/dsview-core/src/lib.rs, crates/dsview-sys/src/lib.rs, crates/dsview-sys/bridge_runtime.c</files>
  <action>Make the core acquisition flow explicitly own stop-if-running and release-on-exit behavior. Ensure the run path attempts `ds_stop_collect()` when a run is still active or timed out, always follows with device release, and reports partial cleanup failures without abandoning the release attempt. Use RAII or equivalent guard patterns so early returns cannot skip cleanup.</action>
  <acceptance_criteria>
    - start failure after config apply still ends in a release attempt
    - run failure, detach, or timeout triggers stop before release when collection is still active
    - successful runs still verify that the device is no longer collecting before release completes
    - cleanup sequencing is centralized enough that future run paths cannot easily bypass it
  </acceptance_criteria>
</task>

<task type="auto">
  <name>Make bounded wait and cancellation policy explicit for v1</name>
  <read_first>
    - .planning/phases/04-acquisition-execution/04-RESEARCH.md
    - crates/dsview-core/src/lib.rs
    - crates/dsview-cli/src/main.rs
  </read_first>
  <files>crates/dsview-core/src/lib.rs, crates/dsview-cli/src/main.rs</files>
  <action>Choose and implement a minimal bounded-wait strategy so a broken hardware or session state cannot hang the CLI indefinitely. Whether the timeout is internal-only or exposed as a narrow run option, define how expiry is detected, how stop and release are triggered, and how the resulting failure is reported to the caller.</action>
  <acceptance_criteria>
    - the acquisition command has a documented policy for runs that never reach a terminal event
    - timeout expiry leads to explicit stop and release attempts rather than process abandonment
    - timeout behavior is reported as a specific non-zero failure outcome
    - the decision preserves Phase 4 scope and does not require a full interactive cancel UX
  </acceptance_criteria>
</task>

<task type="auto">
  <name>Attach a verification method to every promised failure class</name>
  <read_first>
    - .planning/phases/04-acquisition-execution/04-RESEARCH.md
    - .planning/STATE.md
    - crates/dsview-core/src/lib.rs
    - crates/dsview-cli/src/main.rs
  </read_first>
  <files>.planning/phases/04-acquisition-execution/04-RESEARCH.md, crates/dsview-core/src/lib.rs, crates/dsview-cli/src/main.rs</files>
  <action>For each failure class promised by Phase 4, define how it will be verified and where that proof lives. Use synthetic tests for result mapping where possible, environment-gated local checks for preflight-blocked behavior, and manual hardware steps only where real DSLogic behavior is required. Document the split so reviewers can see that every category has a concrete validation path.</action>
  <acceptance_criteria>
    - `start_failure` has a defined verification path, such as a synthetic bridge/core outcome or an environment-gated local start refusal case
    - `run_failure` has a defined verification path, such as a synthetic terminal error event mapped through the core layer
    - `detach` has a defined verification path, such as a synthetic event-path test plus a manual hardware unplug check if safely feasible
    - `incomplete` has a defined verification path, such as synthetic summaries missing required end or logic-packet signals
    - `timeout` has a defined verification path, such as a synthetic non-terminal wait outcome driving stop and release logic
    - `cleanup_failure` has a defined verification path, such as injected stop or release failure in tests
    - `preflight_blocked` has a defined verification path, such as local permission or resource readiness checks on this machine
  </acceptance_criteria>
  <gate>
    - Do not mark this plan complete unless every failure category named in the plan is paired with a verifier-ready method.
  </gate>
</task>
