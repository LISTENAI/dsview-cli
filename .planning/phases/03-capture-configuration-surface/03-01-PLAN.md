<objective>
Define the Rust-side request, capability, validation, and effective-configuration types needed to support `DSLogic Plus` capture configuration before acquisition starts.
</objective>

<scope>
This plan is limited to domain modeling and validation behavior in `dsview-core`. It does not add native config writes, acquisition start/stop, or export logic.
</scope>

<task type="auto">
  <name>Define capture configuration request and capability types</name>
  <read_first>
    - .planning/phases/03-capture-configuration-surface/03-RESEARCH.md
    - .planning/ROADMAP.md
    - .planning/REQUIREMENTS.md
    - crates/dsview-core/src/lib.rs
    - crates/dsview-core/src/session.rs
    - crates/dsview-sys/src/lib.rs
  </read_first>
  <files>crates/dsview-core/src/capture_config.rs, crates/dsview-core/src/lib.rs</files>
  <action>Create Rust domain types for a requested capture configuration, a device capability snapshot, and a validated/effective capture configuration. The model must represent sample rate, sample limit/depth, and enabled logic channels directly, while leaving room for internal mode-derived details required for validation.</action>
  <acceptance_criteria>
    - a request type exists for CAP-01, CAP-02, and CAP-03 inputs
    - a capability type exists for samplerates, valid-channel ceilings, depth limits, and any mode-dependent constraints needed for CAP-04
    - an effective configuration type exists to represent normalized values after validation, including any aligned sample limit/depth
    - domain types do not expose raw pointers, GLib values, or DSView-private ownership details
  </acceptance_criteria>
</task>

<task type="auto">
  <name>Implement DSLogic Plus capture validation rules</name>
  <read_first>
    - .planning/phases/03-capture-configuration-surface/03-RESEARCH.md
    - crates/dsview-core/src/error.rs
    - upstream notes on SR_CONF_SAMPLERATE, SR_CONF_LIMIT_SAMPLES, SR_CONF_CHANNEL_MODE, SR_CONF_PROBE_EN, SR_CONF_VLD_CH_NUM, and SR_CONF_HW_DEPTH
  </read_first>
  <files>crates/dsview-core/src/capture_config.rs, crates/dsview-core/src/error.rs</files>
  <action>Implement validation that checks requested settings against the active device capability snapshot. Cover unsupported sample rates, invalid or empty channel selections, enabled-channel ceilings, sample-limit/depth constraints per enabled-channel count, and normalization/alignment behavior where upstream rounds or constrains requested values.</action>
  <acceptance_criteria>
    - unsupported sample rates are rejected with a specific validation error
    - invalid channel selections are rejected before acquisition begins
    - sample limit/depth validation accounts for enabled-channel-dependent ceilings
    - alignment or normalization behavior is captured explicitly in the effective configuration instead of being hidden
    - validation errors distinguish malformed requests from unsupported device-specific combinations
  </acceptance_criteria>
</task>

<task type="auto">

  <read_first>

    - crates/dsview-sys/src/lib.rs

  <files>crates/dsview-core/src/session.rs, crates/dsview-core/src/lib.rs</files>
  <action>Shape the core API so an opened `DSLogic Plus` session can load capabilities, validate a requested capture config, and obtain a validated/effective configuration object that later phases can apply and reuse. Make it explicit that acquisition cannot begin until this validation step succeeds.</action>
  <acceptance_criteria>
    - the session API has an explicit pre-run validation step
    - the output of validation is reusable by later native-apply and acquisition work
    - the contract makes it difficult to skip validation accidentally
    - the design remains limited to config only and does not imply acquisition start behavior
  </acceptance_criteria>
</task>
