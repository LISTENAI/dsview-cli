<objective>
Extend the Phase 2 active-device/session bridge so `dsview-core` can load capture capabilities from an opened `DSLogic Plus` and apply only validated capture settings before acquisition starts.
</objective>

<scope>
This plan covers native capability reads and config application only. It does not add CLI UX, acquisition execution, or export behavior.
</scope>

<task type="auto">
  <name>Expose capability reads through the narrow sys boundary</name>
  <read_first>
    - .planning/phases/03-capture-configuration-surface/03-RESEARCH.md
    - crates/dsview-sys/src/lib.rs
    - crates/dsview-sys/wrapper.h
    - native shim source from Phase 2
    - DSView facade/config functions for active-device configuration
  </read_first>
  <files>crates/dsview-sys/wrapper.h, crates/dsview-sys/native shim source, crates/dsview-sys/src/lib.rs</files>
  <action>Add the smallest ABI extension needed to fetch the active device's capture capabilities for Phase 3 validation. Normalize samplerates and other required validation helpers into Rust-safe shapes before they leave `dsview-sys`, and keep pointer-valued list handling fully contained in native code.</action>
  <acceptance_criteria>
    - samplerate discovery is exposed as a Rust-safe list of numeric values
    - enabled-channel ceilings, hardware-depth information, and other required validation helpers are available without exposing native ownership details
    - pointer-valued native list handling does not leak into `dsview-core`
    - the sys boundary remains scoped to active-device configuration only
  </acceptance_criteria>
</task>

<task type="auto">
  <name>Apply validated sample rate, limit, and channel settings</name>
  <read_first>
    - crates/dsview-core/src/capture_config.rs
    - crates/dsview-core/src/session.rs
    - crates/dsview-sys/src/lib.rs
    - DSView config setters and channel enable helpers
  </read_first>
  <files>crates/dsview-sys/native shim source, crates/dsview-sys/src/lib.rs, crates/dsview-core/src/session.rs</files>
  <action>Implement the session-layer path that takes a validated/effective capture configuration and applies it to the active device using the existing DSView config facade. Use `ds_set_actived_device_config` for supported scalar settings and `ds_enable_device_channel_index` for per-channel enable state. Keep ordering explicit if upstream coupling requires channels or mode-derived state to be applied before final rate/limit values.</action>
  <acceptance_criteria>
    - sample rate can be applied from a validated/effective configuration
    - sample limit/depth can be applied from a validated/effective configuration
    - enabled channels can be applied deterministically by channel index
    - the application path accepts only validated/effective config types, not raw user requests
    - any native apply failure is surfaced as a stable core error without starting acquisition
  </acceptance_criteria>
</task>

<task type="auto">
  <name>Define pre-run configuration lifecycle in the session API</name>
  <read_first>
    - crates/dsview-core/src/session.rs
    - crates/dsview-core/src/error.rs
    - crates/dsview-core/src/capture_config.rs
  </read_first>
  <files>crates/dsview-core/src/session.rs, crates/dsview-core/src/error.rs</files>
  <action>Refine the core session API so callers can: load capabilities from an opened device, validate a request, apply the validated settings, and receive either a stable success result or an actionable failure before any acquisition path is entered. Document whether config application mutates session state and what later phases may rely on as the configured baseline.</action>
  <acceptance_criteria>
    - the session API has a clear ordered flow: open -> capabilities -> validate -> apply
    - config application is impossible without an opened supported device session
    - failure paths leave the session reusable for a corrected configuration attempt or clean release
    - no acquisition start or stop behavior is added in this plan
  </acceptance_criteria>
</task>
