<objective>
Add focused automated coverage for Phase 3 so valid `DSLogic Plus` capture settings pass, invalid or unsupported combinations fail early, and the native config bridge remains honest about environment- and device-specific limits.
</objective>

<scope>
This plan covers tests and validation evidence for capture configuration only. It does not test acquisition execution or export output.
</scope>

<task type="auto">
  <name>Add core validation tests for accepted and rejected configurations</name>
  <read_first>
    - crates/dsview-core/src/capture_config.rs
    - crates/dsview-core/src/error.rs
    - .planning/phases/03-capture-configuration-surface/03-RESEARCH.md
    - .planning/REQUIREMENTS.md
  </read_first>
  <files>crates/dsview-core/tests/capture_config.rs</files>
  <action>Create table-driven tests around capability snapshots and requested configs. Cover valid sample rate/limit/channel combinations, unsupported sample rates, invalid channel selections, empty channel sets if disallowed, over-depth requests, and normalization/alignment of sample limits where upstream behavior requires it.</action>
  <acceptance_criteria>
    - tests cover CAP-01, CAP-02, and CAP-03 through the core request model
    - tests cover CAP-04 by asserting early rejection of invalid or unsupported combinations
    - at least one success case asserts the effective configuration differs from the request only when normalization/alignment is expected
    - validation errors are asserted by category, not only by free-form message text
  </acceptance_criteria>
</task>

<task type="auto">

  </read_first>
  <files>crates/dsview-core/tests/capture_config.rs</files>
  <action>Add targeted tests that prove the validation layer understands DSLogic Plus-specific coupling. Include cases where changing the number of enabled channels changes the maximum valid depth, and cases where a sample rate is valid only under certain capability-derived conditions.</action>
  <acceptance_criteria>
    - at least one test demonstrates enabled-channel count changing the validity of a requested depth
    - at least one test demonstrates a device-specific coupled constraint beyond simple range checking
    - test fixtures remain explicit enough for future maintainers to update when upstream device behavior changes
  </acceptance_criteria>
</task>
