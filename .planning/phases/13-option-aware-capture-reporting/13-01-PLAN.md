<objective>
Apply the full validated DSLogic Plus option request through the runtime before acquisition starts, and preserve truthful partial-apply failure facts when the device rejects a step.

Purpose: Phase 12 already resolves and validates the option request, but Phase 13 must actually drive the hardware in a deterministic way without falling back to the current active-mode assumptions.
Output: A sys/core/cli execution path that consumes the validated request, applies it in the locked order, fails fast on the first runtime rejection, and keeps the pre-Phase-13 baseline when no option-aware request is present.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Phase 11 validated request -> runtime setters | A request that was safe in pure validation must still be applied in the exact locked order without hidden re-interpretation |
| Runtime setters -> partially configured device | A failing setter can leave hardware in an intermediate state that must be reported honestly |
| CLI capture command -> core execution path | The CLI must pass the validated request into core without discarding it or rebuilding it from raw strings |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01-01 | T | `crates/dsview-core/src/lib.rs` option-aware apply helper | mitigate | Consume only `ValidatedDeviceOptionRequest`, apply the exact D-05 order in Rust, and keep the old config-only path separate so explicit user intent is not rewritten |
| T-13-01-02 | R | `crates/dsview-core/src/lib.rs` partial-apply failures | mitigate | Record `applied_steps`, `failed_step`, and the underlying runtime error in a typed failure contract that later CLI rendering can serialize directly |
| T-13-01-03 | D | `crates/dsview-sys/bridge_runtime.c` mock/runtime bridge | mitigate | Add fail-fast order tests and failure injection so hidden retries or out-of-order setter calls are caught before hardware runs |
</threat_model>
