<objective>
Replace the old explicit runtime-selection UX with one truthful Phase 1 command contract that auto-discovers the bundled runtime/resources and preserves only the documented resource override.
</objective>

<scope>
This plan covers `dsview-core` discovery policy, CLI flag removal, developer fallback behavior for `cargo run`, and command-boundary tests that lock the public surface before README and automation are written.
</scope>

  <read_first>

  <files>crates/dsview-core/src/lib.rs, crates/dsview-core/tests/bundle_discovery.rs</files>
  <action>Refactor `crates/dsview-core/src/lib.rs` so runtime discovery becomes an internal policy instead of a caller-provided choice. Add helpers that first resolve `<exe_dir>/runtime/<platform-runtime-name>` and `<exe_dir>/resources`, then fall back to repo-local developer locations for `cargo run` only when the bundle-relative paths do not exist. Keep `--resource-dir` as an explicit override that replaces only resource directory resolution, and ensure `ResourceDirectory::discover()` still validates the DSLogic Plus file set after resolution.</action>
  <acceptance_criteria>
    - `dsview-core` exposes a default runtime/resource discovery path that does not require CLI callers to pass `--library` or `--use-source-runtime`
    - the default runtime path is derived from the executable directory and the target-correct runtime filename
    - the default resource path is derived from the executable directory as `resources/`
    - explicit resource override precedence is preserved without making runtime location user-configurable again
  </acceptance_criteria>
</task>
