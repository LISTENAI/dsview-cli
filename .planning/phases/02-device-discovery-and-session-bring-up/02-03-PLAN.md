<objective>
Expose the Phase 2 device-discovery and session bring-up workflow through the CLI so users can list supported hardware, select `DSLogic Plus`, and receive clear diagnostics when discovery or open fails.
</objective>

<scope>
This plan covers CLI UX and diagnostics only for listing and bring-up/open validation. It does not start capture, apply acquisition configuration, or produce export artifacts.
</scope>

<task type="auto">
  <name>Design the narrow Phase 2 CLI surface</name>
  <read_first>
    - crates/dsview-cli/src/main.rs
    - crates/dsview-core/src/lib.rs
    - .planning/ROADMAP.md
    - .planning/REQUIREMENTS.md
  </read_first>
  <files>crates/dsview-cli/src/main.rs, crates/dsview-cli/src/commands.rs</files>
  <action>Add the smallest scriptable command surface that satisfies Phase 2: a list command for supported devices and a bring-up/open command that targets one supported device without starting capture. Keep arguments explicit and automation-friendly. Define a deterministic selector contract for multiple supported devices, such as an explicit handle/index plus a stable machine-readable identifier in list output.</action>
  <acceptance_criteria>
    - the CLI exposes a supported-device list path
    - the CLI exposes a way to target one supported device for open/close bring-up
    - the selector contract is deterministic when multiple supported devices are attached
    - command names and arguments stay compatible with later capture-focused expansion
  </acceptance_criteria>
</task>

  <action>Render device-list results and open/close outcomes in a scriptable format, and normalize core errors into actionable CLI diagnostics. Define a concrete machine-readable output contract for list and bring-up failure paths, and cover common failure modes such as no supported device, unsupported selection, firmware/resource setup issues, and native open failure.</action>
  <acceptance_criteria>

    - exit behavior is explicit and non-zero on bring-up failure paths
  </acceptance_criteria>
</task>
