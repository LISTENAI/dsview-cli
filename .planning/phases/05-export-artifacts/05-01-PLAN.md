<objective>
Retain the capture packet stream needed for export and route it through the upstream VCD output path so a clean `DSLogic Plus` capture can produce a valid VCD without modifying the read-only `DSView/` submodule.
</objective>

<scope>
This plan covers packet retention, ordered replay, export lifecycle ordering, VCD temp-file/finalization semantics, and the minimal core integration needed to request VCD export after a clean capture. It does not define the final metadata schema or the full artifact validation matrix from later plans.
</scope>

  <acceptance_criteria>
    - the sys layer looks up the `vcd` output module and fails cleanly when it is unavailable
    - replay reconstructs packets accepted by `sr_output_send()` for `SR_DF_META`, `SR_DF_LOGIC`, and `SR_DF_END`
    - non-OK end-packet status or missing clean-success stream is rejected before writing a v1-valid VCD
    - capacity-overflow streams are rejected before export begins and are distinguishable from native exporter failures
    - the sys export API writes to a temporary path first and only promotes to the final VCD path on clean success, or removes the temporary file if promotion cannot occur
    - the sys export API returns stable export facts needed by higher layers, including actual sample count and output byte or packet counts when available
    - VCD export relies on upstream naming and timing semantics rather than recreating them in Rust
  </acceptance_criteria>
</task>
