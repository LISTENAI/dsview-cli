<objective>
Turn the Phase 5 artifact contract into verifier-ready proof by adding deterministic goldens, normalized semantic checks, and manual hardware validation for exported VCD and metadata outputs.
</objective>

<scope>
This plan covers automated golden-file validation, schema and CLI contract checks, Nyquist-aware timing guidance, and real-hardware export UAT. It does not redesign the export or metadata implementation from the earlier plans.
</scope>

  <name>Create deterministic synthetic packet fixtures and VCD golden files</name>
  <read_first>
    - crates/dsview-sys/tests/boundary.rs
    - .planning/phases/05-export-artifacts/05-RESEARCH.md
    - .planning/phases/05-export-artifacts/05-VALIDATION.md
  </read_first>
  <files>crates/dsview-sys/tests/boundary.rs, crates/dsview-sys/tests/goldens or equivalent fixture directories</files>
  <action>Build tiny synthetic packet streams that exercise the same sys replay bridge used in production export, then compare the produced VCD output against checked-in goldens. Keep fixtures small enough for manual inspection and deliberate enough to prove channel naming, initial values, delta transitions, final timestamps, packet-boundary replay behavior, and one explicit retention-capacity edge.</action>
  <acceptance_criteria>
    - fixtures include at least one 1-channel or 2-channel stream with explicit `SR_DF_META`, short `SR_DF_LOGIC` packets, and a closing `SR_DF_END`
    - checked-in VCD goldens assert `$timescale`, `$var` declarations, initial value emission, delta-only transitions, and final timestamp presence
    - the test path uses the production sys replay bridge rather than a separate test-only serializer
    - fixture names and golden files make the expected timing and channel semantics obvious to reviewers
    - at least one fixture or companion assertion exercises the retention-capacity boundary or overflow-precondition path explicitly
  </acceptance_criteria>
</task>

<task type="auto">
  <name>Add normalized semantic assertions for timing, sample counts, metadata structure, and cleanup-safe writes</name>
  <read_first>
    - crates/dsview-sys/tests/boundary.rs
    - crates/dsview-core/tests/export_artifacts.rs
    - .planning/phases/05-export-artifacts/05-RESEARCH.md
  </read_first>
  <files>crates/dsview-sys/tests/boundary.rs, crates/dsview-core/tests/export_artifacts.rs</files>
  <action>Complement full-file goldens with assertions that normalize away irrelevant formatting while preserving semantic proof. Parse or inspect exported VCD and metadata into event- or field-level checks so timing relationships, sample counts, required JSON shape, and final-path cleanup behavior remain protected even if harmless formatting changes later occur.</action>
  <acceptance_criteria>
    - sys-level checks verify transition times are consistent with the fixture sample rate and known sample sequence
    - sys-level checks verify actual sample count matches the retained packet data used for replay
    - sys-level checks verify overflow or export failure paths do not leave a misleading final-path VCD artifact
    - core-level checks verify required metadata fields, numeric typing, and stable schema structure without depending on wall-clock timestamps or absolute paths
    - core-level checks verify metadata finalization occurs only after VCD success and that failed metadata writes do not leave a misleading final-path sidecar
    - normalized checks remain aligned with the same production export and metadata code paths used outside tests
  </acceptance_criteria>
</task>
