<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `dsview-core` option snapshot -> `dsview-cli` token layer | Stable IDs and labels come from trusted discovery code, but the CLI must not lose the authoritative stable IDs when adding user-facing tokens |
| `devices options` JSON/text -> shell and agent consumers | Inspection output becomes the copy/paste source for future `capture` flags, so drift or ambiguity will misconfigure later commands |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01-01 | T | `crates/dsview-cli/src/capture_device_options.rs` token generation | mitigate | Generate deterministic tokens from stable label/ID pairs, keep stable IDs in every response object, and lock the contract with `device_options_cli` JSON assertions |
| T-12-01-02 | R | `crates/dsview-cli/src/device_options.rs` text renderer | mitigate | Render explicit flag names and current values so copy/paste guidance is auditable and test-locked rather than implied |
| T-12-01-03 | I | `devices options` machine-readable output | mitigate | Preserve device identity and stable IDs in JSON so automation does not regress when friendly tokens are added |
</threat_model>
